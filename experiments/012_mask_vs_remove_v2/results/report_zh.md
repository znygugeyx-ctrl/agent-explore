# 实验 012：Mask vs Remove 策略对比分析报告

**模型：** Qwen3-8B / Qwen3-14B
**日期：** 2026-03-29
**任务数：** 15 个（3–10 轮对话）
**工具数：** 60 个（12 组 × 5 个/组）
**每策略总轮次：** 93 轮

---

## 一、实验背景

### 核心假设

本实验验证两个独立假设：

- **H1（准确率）**：对每轮对话动态限制可用工具集，是否能提升模型在大工具池中选择正确工具的准确率？

- **H2（缓存命中率 / TTFT）**：Manus 提出的"Mask, Don't Remove"核心主张——保持 prompt prefix（系统提示 + 工具定义）稳定，能否保留 KV 缓存、降低首 token 延迟（TTFT）？

### 四种策略

| 策略 | 工具展示 | 每轮变化？ | Prefix 稳定？ |
|---|---|---|---|
| `all` | 全部 60 个工具，无限制 | ❌ | ✅ 基准 |
| `remove_dynamic` | 每轮仅展示当前组 5 个工具 | ✅ | ❌ 每轮重建 |
| `mask_hint` | 60 个工具 + 用户消息前注入可用工具名 | ❌ | ✅ |
| `mask_logit` | 60 个工具 + 组级 logit 封锁不可用组 | ❌ | ✅ |

---

## 四、H2 验证：KV 缓存命中率

### 结论：H2 **强力支持** ✅，14B 上效果更显著

**Qwen3-8B 缓存数据：**

```
策略              缓存命中率    查询 tokens    命中 tokens
all               99.8%        454,038       453,056
mask_hint         98.9%        457,258       452,080
mask_logit        98.8%        448,050       442,592
remove_dynamic    75.1%         81,719        61,376
```

**Qwen3-14B 缓存数据：**

```
策略              缓存命中率
all               97.2%
mask_hint         96.8%
mask_logit        98.2%
remove_dynamic    71.0%
```

**核心发现：**

1. **工具定义稳定 → 缓存命中率 ~97–100%**：`all`、`mask_hint`、`mask_logit` 在两个模型上均维持高命中率，60 个工具定义的大 prefix 被完整缓存复用。

2. **`remove_dynamic` 缓存损失在 14B 上更大**：8B 损失 ~25pp（99.8% → 75.1%），14B 损失 ~26pp（97.2% → 71.0%）。14B 工具定义编码更复杂（模型更大、注意力层更多），每轮重建 prefix 的代价更高。

3. **输入 tokens 决定缓存空间**：`remove_dynamic` 每轮仅传 5 个工具（平均 879 tokens），远少于其他策略（~4,900 tokens）。虽然单次请求更轻量，但失去缓存复用的收益远大于减少 tokens 的节省。

### Qwen3-8B

```
logit      0.670  0.628  0.641  0.636  0.670  0.673  0.667  0.735  0.699  0.708
```

### Qwen3-14B

```
轮次       t0     t1     t2     t3     t4     t5     t6     t7     t8     t9
all        0.991  0.751  0.743  0.776  0.744  0.813  0.780  0.770  0.869  0.762
rdyn       0.776  0.757  0.731  0.735  0.759  0.749  0.770  0.755  0.760  0.786
hint       0.895  0.770  0.772  0.775  0.777  0.749  0.841  0.798  0.796  0.787
logit      0.769  0.715  0.750  0.734  0.758  0.747  0.765  0.805  0.774  0.786
```

**关键观察：**

- **8B `remove_dynamic` 在 t7（0.954s）、t8（1.049s）出现明显尖峰**：10 轮长任务末尾，每轮重建工具 prefix + 积累的历史消息使计算量线性增长，TTFT 翻倍（0.65s → 1.05s）。

- **14B `all` 在 t0 最高（0.991s），t1 后急降至 ~0.75s**：14B 的大 prefix（60 工具 ~6K tokens + 更大模型权重）在冷启动时更重，但一旦缓存命中，后续 TTFT 反而比 8B 更稳定——14B 缓存效益的绝对值更大。

- **14B `remove_dynamic` 无明显 TTFT 尖峰**（t7: 0.755s，t8: 0.760s）：与 8B 不同，14B 的 `remove_dynamic` TTFT 整体较平。原因在于 14B 处理 5 个工具的小 prefix 也较慢（基础推理速度更低），掩盖了缓存缺失的惩罚；但 75.1% 的缓存命中率损失在实际 GPU 利用率和吞吐量上仍有显著代价。

- **整体规律**：8B TTFT 基准 ~0.65s，14B TTFT 基准 ~0.75s，差距约 0.1s，符合模型规模增大的预期。

## 六、策略综合评价

### `mask_hint`：最优综合策略 ⭐

| | 8B | 14B |
|---|---|---|
| 准确率 | 98.9% | 98.9% |
| 缓存命中 | 98.9% | 96.8% |
| 副作用 | 无 | 无 |

- 工具定义完全不动，系统 prompt 不变 → prefix cache 完整保留
- 每轮用户消息前追加一行：`[Available tools this turn: math_add, math_gcd, ...]`
- **两个模型上准确率均最优（98.9%）**，与 `remove_dynamic` 并列，无任何副作用
- **最贴近真实场景**（MCP routing hint 风格）：路由层在转发请求时附加上下文，工具注册表不变

### `mask_logit`：有效但有隐患

| | 8B | 14B |
|---|---|---|
| 准确率 | 98.9% | 97.2% |
| 缓存命中 | 98.8% | 98.2% |
| 副作用 | 轻微 | 更明显 |

- logit bias 封锁所有输出 token，**包括 `<think>` 思维链段落**
- 14B 上准确率已下降至 97.2%（与 `all` 基准持平），说明思维链干扰在更大模型上更显著——14B 更依赖显式内部推理，封锁工具组名后推理质量有所下降
- 缓存命中率与 `mask_hint` 相当，但精度代价更高，**不推荐优先使用**

### `remove_dynamic`：准确但缓存昂贵

| | 8B | 14B |
|---|---|---|
| 准确率 | 98.9% | 98.9% |
| 缓存命中 | 75.1% | 71.0% |
| 副作用 | 长对话 TTFT 尖峰（8B 明显） |

- 准确率在两个模型上均为最高（98.9%），工具集小，干扰低
- 缓存命中率最差（~71–75%），随模型规模增大损失略增
- 高并发场景下 GPU KV cache 压力更大，整体吞吐下降

### `all`：基准，适合简单场景

| | 8B | 14B |
|---|---|---|
| 准确率 | 98.1% | 97.2% |
| 缓存命中 | 99.8% | 97.2% |
| 副作用 | 无 |

- 最简单实现，缓存效率最高
- 准确率略低（0.8–1pp），在长任务中偶尔因无引导出现工具混淆

## 七、主要结论

### 结论 1：Manus 假设在两个模型上均得到验证，14B 上缓存损失更大

`remove_dynamic` 使 KV 缓存命中率从 ~99% 降至 71–75%，损失约 25pp。14B（71.0%）比 8B（75.1%）损失更多，原因是更大模型的工具定义编码代价更高。**模型越大，保持 prefix 稳定的收益越显著。**

### 结论 2：mask_hint 是工程实践中的最优解，在两个模型上稳定最优

| 指标 | `mask_hint` vs `remove_dynamic` | `mask_hint` vs `mask_logit` |
|---|---|---|
| 准确率 | 持平（98.9%） | 8B 持平，14B 高 1.7pp |
| 缓存命中 | 高 ~26pp | 持平 |
| 副作用 | 无 vs 长对话延迟 | 无 vs 思维链干扰 |

### 结论 3：H1 在当前任务规模下不显著，与模型大小无关

60 工具对 8B 和 14B 均不构成选择困难。工具限制带来的准确率提升不超过 1pp，且唯一一致失败的任务（`chain_M_T_U_C_M_T`，83%）在所有策略和模型上均失败，确认失败原因是任务难度而非工具策略。

## 八、局限性与后续方向

| 局限 | 说明 |
|---|---|
| 单次运行 | 1 run × 15 tasks，统计显著性有限；生产结论需 3 次以上运行 |
| 合成任务 | 提示场景化但任务确定性高，真实工具调用噪音更大 |
| 未覆盖弱模型 | 1-4B 模型的 H1 效果可能更显著（工具池大小对弱模型影响更大） |
| TTFT 波动 | vLLM 调度和批处理引入 ±50ms 抖动，逐轮 TTFT 分析需更大样本 |
| 并发未测试 | 单请求串行测试，未反映高并发下缓存竞争的放大效应 |

**后续实验方向：**
1. **弱模型（1.7B / 4B）**：验证工具池大小对准确率的影响是否在小模型上显著放大
2. **高并发测试**：并发 20–50 请求，量化缓存命中率差异对实际吞吐和 GPU 利用率的影响
3. **真实工具集**：用 BFCL benchmark 真实工具替换合成工具，验证复杂描述下的缓存命中率
