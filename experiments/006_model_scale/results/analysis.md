# 实验 006：Guided Decoding × 模型规模 — 分析报告

**日期**：2026-03-24
**模型**：Qwen3-0.6B / 1.7B / 4B / 8B / 14B via vLLM 0.18.0（/no_think 模式，temperature=0.0）
**规模**：30 个任务 × 3 个策略 × 3 轮 × 5 个模型 = 1350 次 LLM 调用
**注**：Qwen3-32B 通过 Docker + g6e.12xlarge（4× L40S 192GB）以 TP=4 运行，max_model_len=8192

---

## 一、核心数据

### H1：结构正确性 vs 模型规模

| 模型 | prompt parse | guided parse | 关键含义 |
|------|-------------|-------------|---------|
| 0.6B | **0%** | **100%** | prompt 完全失效，guided 是唯一可用方案 |
| 1.7B | 100% | 100% | 两者均可靠 |
| 4B   | 100% | 100% | 两者均可靠 |
| 8B   | 100% | 100% | 两者均可靠 |
| 14B  | 100% | 100% | 两者均可靠 |
| 32B  | 100% | 100% | 两者均可靠 |

**0.6B 是分水岭**：该模型完全无法通过 prompt 指令生成 JSON，1.7B 是指令跟随能力的硬断点。

---

## 二、核心假设验证

### H1：更小的模型 prompt parse 率更低 ✅ 强烈确认（存在硬断点）

阶梯不是渐变的：0.6B 完全无法遵循 JSON schema 指令（0%），1.7B 及以上全部 100%。

---

### H2：Guided 净收益（guided - prompt）随模型缩小而增大

| 模型 | B1 role guided | B1 role prompt | guided_benefit | A role 清晰 benefit |
|------|---------------|----------------|----------------|---------------------|
| 0.6B | 50% | N/A (0% parse) | — | — |
| 1.7B | 25% | 75%  | **-50pp** | -10pp |
| 4B   | 0%  | 50%  | **-50pp** | -60pp |
| 8B   | 0%  | 75%  | **-75pp** | -40pp |
| 14B  | 75% | 100% | **-25pp** | -10pp |
| 32B  | 50% | 50%  | **0pp**   | -10pp |

**32B 新发现**：guided_benefit 首次达到 **0pp**（50% vs 50%），是曲线的新最高点。但注意 prompt 在 32B 上只有 50%（低于 14B 的 100%），说明 32B 在 B1 role 歧义上 prompt 本身表现退步，而非 guided 有改善。guided 与 prompt 打平，但两者都不如 14B。

**但 guided 在所有规模上都未能超过 prompt**——H2（guided 小模型收益更大）从"语义净收益"角度来看仍然不成立。

---

### H3：Enum 偏置强度与模型规模的关系 ⚠️ 非单调

| 模型 | A清晰 role (guided) | B1歧义 role (guided) |
|------|--------------------|--------------------|
| 0.6B | 90% | 50% |
| 1.7B | 90% | 25% |
| 4B   | **30%** | 0% |
| 8B   | 60% | 0% |
| 14B  | **90%** | 75% |
| 32B  | 90% | 50% |

- 4B 的 enum 偏置最严重（A 清晰 role 仅 30%）——非单调现象，可能与该模型预训练数据中 `designer` token 频率相关
- 14B 在 B1 歧义上 guided 表现最佳（75%），32B 次之（50%）
- 整体趋势是更大模型偏置更小，但 4B 是明显的异常点，32B 的 B1 guided=50% 介于 14B 与 1.7B 之间（非单调）

|------|-------------------|-------------------|

| 8B   | 100% 幻觉 | 100% 幻觉 |
| 14B  | 100% 幻觉 | 100% 幻觉 |
| 32B  | 100% 幻觉 | 100% 幻觉 |

32B 延续 4B/8B/14B 模式：两种策略均 100% 幻觉。

### H6：延迟对比

| 模型 | prompt 延迟 | guided 延迟 | 差值 |
|------|------------|------------|------|
| 1.7B | 1.16s | 1.22s | +0.06s (+5%) |
| 4B   | 1.65s | 1.68s | +0.03s (+2%) |
| 8B   | 2.20s | 2.44s | +0.24s (+11%) |
| 14B  | 3.14s | 3.01s | **-0.13s (-4%)** |
| 32B  | 3.37s | 2.95s | **-0.42s (-12%)** |

**32B 的延迟结论**：guided 比 prompt 快 0.42s（-12%），是所有规模中最大的 guided 延迟优势。32B 的 guided 输出 token 数（51）少于 prompt（62），TP=4 多卡并行下 xgrammar 开销可以忽略。

## 三、最重要的发现（含 14B）

### 发现 1：0.6B 指令跟随断点（不变）
0.6B 完全无法生成 JSON，1.7B 是硬断点。这对工程部署有强约束力。

### 发现 2：32B guided_benefit 首次达到 0pp，但 prompt 本身也退步
32B guided=50%，prompt=50%，benefit=0pp——曲线上的首次打平。但这不是因为 guided 改善，而是 32B 的 prompt 在 B1 role 歧义上退步到 50%（低于 14B 的 100%）。guided 与 prompt 打平但两者都不如 14B prompt 的 100%。

### 发现 3：4B 是 enum 偏置最严重的异常点
4B 在 A 清晰 role 上仅 30%（guided），远低于相邻规模的 1.7B（90%）和 8B（60%）。这一非单调现象表明 enum 偏置不是简单的规模函数。

### 发现 4：论文 H7 在全部 5 个规模上成立
"强约束是可靠性增强器，不是能力增强器"——在所有可比较规模（1.7B–14B）上，guided 从未在语义准确率上超过 prompt。

---

## 四、B1 role 歧义准确率趋势（核心对比图）

```
prompt:  1.7B=75%  4B=50%  8B=75%  14B=100%
guided:  1.7B=25%  4B=0%   8B=0%   14B=75%
benefit: 1.7B=-50pp  4B=-50pp  8B=-75pp  14B=-25pp
```

整体看，更大模型 guided 的语义质量更接近 prompt，但始终低于 prompt。

## 六、工程建议

| 场景 | 建议 |
|------|------|
| 模型 < 1.7B | 必须使用 guided，这是唯一可靠的结构化输出路径 |
| 模型 ≥ 1.7B，schema 含 enum | 优先 prompt-only；guided 损害 enum 准确率（-25pp 到 -75pp）|
| 模型 ≥ 14B，对结构可靠性有强要求 | guided 损失最小（-25pp），可作为额外保险 |
| 模型 = 32B，schema 含 enum | guided 与 prompt 打平（0pp），但两者 B1 role 歧义均只有 50%；若追求 enum 准确率应首选 14B |
| Schema 仅含 string/integer/boolean | guided 与 prompt 效果相当，guided 可作结构保险 |

---

## 七、局限性

1. **单一 schema**：结论高度依赖 enum 字段；纯 string 字段可能得出不同结论
2. **Qwen3 系列特定**：指令跟随断点（~1.7B）在其他模型家族可能不同
3. **32B 基础设施差异**：32B 通过 Docker + g6e.12xlarge（4× L40S, TP=4, max_model_len=8192）运行，与 14B 以下使用裸机不同；延迟和性能在不同部署模式下的可比性有一定限制
4. **4B enum 偏置异常**：需进一步 logit 分析确认机制
5. **/no_think 效果**：小模型（0.6B）可能不理解该后缀，但由于 prompt 和 free 均失败，影响无法隔离
