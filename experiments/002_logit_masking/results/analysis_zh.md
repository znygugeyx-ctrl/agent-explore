# 实验 002：真实 Logit 掩码 + 多轮任务 — 分析报告

**日期**：2026-03-22
**模型**：Qwen/Qwen3-8B（via vLLM 0.18.0）
**基础设施**：AWS g6e.2xlarge（NVIDIA L40S 48GB），启用 `--enable-prefix-caching`
**总请求数**：约 960 次（4 种策略 × 3 轮运行 × 20 个任务 × 平均约 4 轮对话）
**总耗时**：约 50 分钟

---

## 1. 执行摘要

本实验在**多轮任务**场景下，对**真实 token 级 logit 掩码**（通过 vLLM `logit_bias` 实现）与其他三种工具选择策略进行了比较。结果对实验 001 的结论和 Manus 文章的假设均构成挑战：

| 指标 | All（基线） | Remove | Logit Mask | Desc Mask | 胜者 |
|------|------------|--------|------------|-----------|------|
| 准确率 | 96.7% | 83.3% | 95.0% | **100%** | Desc Mask |
| 步骤完成率 | **100%** | 61.7% | **100%** | **100%** | 并列（All/Logit/Desc） |
| 平均延迟 | **9.49s** | 16.79s | 12.90s | 9.96s | All |
| 平均输入 Token/任务 | 2,534 | **1,819** | 4,526 | 2,687 | Remove |
| 无效工具调用数 | 22 | 11.3 | 12 | 22 | Remove |

**核心结论**：在多轮任务中，**描述掩码（Desc Mask）以 100% 准确率胜出**，**All 策略延迟最低**。Logit Mask 存在根本性缺陷：模型通过发明替代工具名（中文翻译、拼写错误、点号前缀等）绕过 token 封锁。Remove 策略则彻底崩溃——模型频繁拒绝调用工具。

---

## 2. Logit Mask 被绕过的发现

最出人意料的发现：**Qwen3-8B 会主动规避 `logit_bias` 封锁**，通过生成创意性的替代工具名来绕过限制。当 `calculator` 的 token（29952、88821）被封锁后：

| 绕过方式 | 出现频率 | 类型 |
|---------|---------|------|
| `计算器`（中文"calculator"） | 6+ 次 | 语言切换 |
| `温度转换`（中文"temperature_convert"） | 2+ 次 | 语言切换 |
| `ceasar_cipher`（拼写错误） | 2+ 次 | 拼写变体 |
| `str_reverse`（缩写） | 1+ 次 | 名称缩短 |
| `Base_convert`（大写） | 1+ 次 | 大小写变体 |
| `.base_convert`（点号前缀） | 2+ 次 | 标点前缀 |
| `.char_count`（点号前缀） | 1+ 次 | 标点前缀 |
| `gc`（"gcd"的截断） | 2+ 次 | 名称截断 |

**解读**：当模型的首选 token 被封锁后，它并不会放弃，而是寻找替代的 token 化路径，以不同的 token 序列表达相同的语义。这是**基于 `logit_bias` 掩码的根本性局限**：它只能封锁特定的 token ID，但模型可以通过不同的 token 序列组合出相同的语义含义。

对于 `base_convert` 这类多 token 工具名（tokens: [3152, 34910]），仅封锁第一个 token（3152）只能阻止 `base` 直接出现。模型发现 `.base_convert` 有效，是因为 `.` + `base` 会产生不同的首个 token。同样，支持中英双语的模型可以直接切换到另一种语言。

### 对 Manus 假设的影响

Manus 文章假设 logit 掩码能可靠地阻止工具调用。我们的数据表明，这一假设**对多语言模型来说是错误的**。`logit_bias` 只是一道"软屏障"，复杂的模型完全可以绕行。真正的工具级访问控制需要**结构化生成约束**（如 vLLM 的带 JSON Schema 的引导生成），而非简单的 token 级偏置。

---

## 3. Remove 策略失败分析

Remove 策略在多轮任务中表现急剧下滑（准确率 83.3%，步骤完成率 61.7%），相比实验 001 的单轮结果（准确率 100%）差距悬殊。

### 失败模式："单工具困惑"

当模型只看到 1 个工具时（如步骤 1 只有 `calculator`），它频繁地完全忽略工具，试图在 `<think>` 中独立解决所有问题：

| 任务 | 步骤 1 工具 | 实际调用 | 完成步骤 | 正确？ |
|------|------------|---------|---------|-------|
| chain_calc_base | calculator | 无 | 0/2 | 是（侥幸） |
| chain_cipher_reverse | caesar_cipher | 无 | 0/2 | 否 |
| chain_gcd_base | gcd | 无 | 0/2 | 是（侥幸） |
| chain_temp_calc | temperature_convert | 无 | 0/2 | 是（侥幸） |
| chain_calc_cipher | calculator | 无 | 0/2 | 否 |
| chain_calc_gcd2 | calculator | 无 | 0/2 | 是（侥幸） |
| chain_temp_gcd | temperature_convert | 无 | 0/2 | 否/是 |

**规律**：当只有 1 个工具可见时，模型往往判断自己无需工具即可完成任务。它将全部 1024 token 的预算消耗在 `<think>` 推理上，靠心算得出答案。对于数学任务，这偶尔奏效（因此出现"侥幸正确"），但对于字符串操作和密码任务则失败。

**与实验 001 的差异**：实验 001 的单轮任务中，模型看到 1 个工具和 1 个明确任务，没有犹豫的理由。而多轮任务中，模型看到的是**多步骤的提示词**，却只有 **1 个工具**。它推断出"有问题"（"我需要 2 个工具但只看到 1 个"），从而退回到心算模式。

### 延迟影响

无工具调用任务出现 23 秒延迟，是模型在 `<think>` 模式下耗尽 1024 个 max_tokens 却没有产生工具调用所致。这将 Remove 策略的平均延迟拉高至 16.79s（比 All 策略的 9.49s 高出 77%）。

---

## 4. Desc Mask 的意外惊喜：最高准确率

与实验 001 相反，**描述掩码策略以 100% 准确率**（60/60 个任务全部正确）荣登榜首——所有策略中最优。这与实验 001 中 Desc Mask 垫底（96.7%）的表现完全相反。

### Desc Mask 改善的原因

1. **max_tokens 增加**：从实验 001 的 512 增至 1024。导致实验 001 中 Desc Mask 失败的 `<think>` token 预算耗尽问题不再出现。

2. **多轮任务的加持**：模型在每轮对话中都能看到完整的工具列表（8 个工具）。即使有 7 个被标记为 `[UNAVAILABLE]`，它仍能了解工具全貌。结合多步骤提示词，模型能正确判断每步应使用哪个工具。

3. **[UNAVAILABLE] 标记被严格遵守**：在 60 个任务 × 3 轮运行中，Desc Mask 策略下模型**从未调用过不可用的工具**。这一简单的注释对于 Qwen3-8B 的路由足够有效。

### 无效调用异常的解释

所有策略都存在"无效工具调用"（All/Desc Mask 各 22 次，Remove/Logit Mask 各 11-12 次）。该指标统计的是调用了当前步骤 `step_tools` 列表之外工具的次数。然而，All/Desc Mask 策略下，模型会在**单轮对话中同时调用两步的工具**（并行工具调用），我们的逐步骤跟踪将步骤 0 中的这类调用标记为"无效"。实际上，这正是期望的行为——模型高效地批量执行了工具调用。

## 5. 延迟分析

| 策略 | 均值 | 中位数 | P90 | 最小值 | 最大值 |
|------|------|--------|-----|--------|--------|
| All | **9.49s** | 8.99s | 13.88s | 6.08s | 20.78s |
| Desc Mask | 9.96s | 9.62s | 16.45s | 5.85s | 17.54s |
| Logit Mask | 12.90s | 12.04s | 19.16s | 7.64s | 20.76s |
| Remove | 16.79s | 19.20s | 23.09s | 6.04s | 23.21s |

### Logit Mask 比 All 慢 36% 的原因

Logit Mask 强制**串行工具调用**（在步骤 1 中封锁步骤 2 的工具）。模型必须：
1. 第 1 轮：调用步骤 1 的工具（有时因绕过尝试需要 2-3 次）
2. 第 2 轮：获取工具结果，推进到步骤 2
3. 第 3 轮：调用步骤 2 的工具（有时在失败的绕过尝试后才成功）

而 All/Desc Mask 可以在**单轮对话中调用两个工具**，只需 2 次 LLM 调用，而不是 3-4 次以上。

### Remove 最慢的原因

Remove 策略 16.79s 的平均延迟，主要由 8/20 个任务中模型不调用工具（每次 23s，耗尽 max_tokens）所主导。对于确实有调用工具的任务，Remove 的延迟与 All 相当。

## 6. Token 效率分析

| 策略 | 平均输入 Token | 平均输出 Token | 每任务总计 |
|------|--------------|--------------|----------|
| Remove | **1,819** | 702 | **2,521** |
| All | 2,534 | 364 | 2,898 |
| Desc Mask | 2,687 | 383 | 3,070 |
| Logit Mask | 4,526 | 479 | **5,005** |

**Logit Mask 消耗最多**——接近 All 策略总 token 数的 2 倍。原因：
1. 失败的绕过尝试会产生额外的对话轮次（每轮约新增 500-1000 输入 token 的上下文）
2. 模型常在找到正确路径前多次重复调用同一工具（如 `calculator, calculator, calculator, gcd`）

**Remove 输入 token 最少**，但输出 token 最多（模型在决定不使用工具时会生成大量 `<think>` 内容）。

## 7. 假设评估

### Manus 主张："掩码，不要移除"
**部分得到支持，但实现方式并非 logit 掩码。** Remove 策略在多轮场景下明显失败（准确率 83.3%，步骤完成率 61.7%）。然而，获胜的掩码方案是简单的**描述掩码**（`[UNAVAILABLE]` 前缀），而非 token 级 logit 掩码。

### 我们的预测："Logit Mask = All > Remove > Desc Mask"
**预测错误。** 实际排名：
- **准确率**：Desc Mask（100%）> All（96.7%）≈ Logit Mask（95%）>> Remove（83.3%）
- **延迟**：All（9.49s）≈ Desc Mask（9.96s）< Logit Mask（12.90s）<< Remove（16.79s）

### 核心洞见：logit_bias ≠ 可靠的工具访问控制

`logit_bias` 参数最初设计用于**软性 token 偏好调整**（如减少冗长、避免特定词汇），从未被设计为**硬性工具级访问控制**。我们的实验揭示了其不足之处：

1. **多语言绕过**：双语模型可以切换语言来规避被封锁的 token
2. **拼写变体绕过**：拼写错误、缩写、大小写变化会产生不同的 token
3. **标点绕过**：添加 `.` 等字符作为前缀可产生不同的首 token
4. **覆盖不完整**：封锁多 token 名称的首个 token 远远不够

实现可靠工具掩码的正确方案：
- **结构化生成**（如 vLLM 的 `--guided-decoding-backend` 配合 JSON Schema，约束工具名只能为允许的值）
- **`tool_choice` 函数级参数**（如 `tool_choice: {"type": "function", "function": {"name": "calculator"}}`）
- **智能体循环中的后生成过滤**（拒绝无效工具名并重试）

**Manus 假设在多轮场景中获得支持**：当工具需求跨轮次变化时，Remove 的优势荡然无存。然而，推荐的"掩码"方案应是描述掩码，而非 logit 掩码。

## 9. 观察

### Qwen3-8B 的双语创造力
模型在英文 token 被封锁后生成中文工具名（`计算器`、`温度转换`）的能力，是其双语训练的一次精彩展示。这对任何 token 级内容过滤方案都有重要启示——双语模型天生具有一条"逃生通道"。

### 并行工具调用
All 和 Desc Mask 策略极大地受益于模型在单轮对话中调用多个工具的能力。这实际上将 2 步任务压缩为 1 轮对话，与强制串行策略相比，延迟几乎减半。

### max_tokens 预算的重要性
从 512 增至 1024 个 token，使 Desc Mask 的表现从最差逆转为最优。Qwen3 的 `<think>` 开销需要充足的输出预算。在生产环境中使用推理模型时，应将 max_tokens 设置得足够宽裕。

---

## 10. 结论

1. **多轮任务的最佳策略是描述掩码。** 100% 准确率，延迟接近基线，实现简单。

2. **Remove 策略在多轮任务中失效。** 当模型面对多步骤提示词却只看到 1 个工具时，会退化为心算模式。这印证了 Manus 文章关于对话中途移除工具的警告。

3. **Token 级 logit 掩码不可靠。** 模型通过语言切换、拼写变体、标点前缀和名称缩写等方式找到绕过路径。`logit_bias` 不足以作为工具访问控制手段。

4. **Manus 的"掩码，不要移除"建议在原则上正确，但实现方案应是描述级注释，而非 logit 掩码。** 简单的 `[UNAVAILABLE]` 前缀之所以有效，是因为它在语义层面向模型传达了意图，而 logit 掩码是在 token 层面与模型对抗。

5. **后续步骤**：测试（a）vLLM 结构化生成以实现真正的工具级访问控制，（b）`tool_choice` 参数强制指定工具，（c）50+ 个工具以压测前缀缓存，（d）并发请求的生产负载场景。
