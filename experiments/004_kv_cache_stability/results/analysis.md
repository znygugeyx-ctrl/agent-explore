# Experiment 004: KV Cache Stability — Analysis

## TL;DR

Manus "围绕KV缓存进行设计" 的两个核心建议通过实验得到验证：

1. **系统提示词中加秒级时间戳 → cache hit rate 从 99.5% 暴跌至 0.3%，TTFT 增加 33%**
2. **截断历史上下文 → 准确率下降，并导致模型陷入无限循环的灾难性失败**

---

## 实验设置

- **模型**: Qwen3-8B on vLLM (EC2 g6e.2xlarge, L40S 48GB)
- **工具**: 50个工具（全部始终展示，工具选择不变）
- **任务**: 20个多步链式任务（4-6步），来自实验003
- **执行**: 顺序（每任务完成后执行下一个，允许跨任务cache预热）
- **Temperature**: 0.0（确定性，减少输出长度对TTFT的干扰）

---

## 结果汇总

| 策略        | 准确率 | 平均延迟 | 平均TTFT | Cache命中率 | 平均输入Token |
|-------------|--------|----------|----------|-------------|---------------|
| stable      | 80.0%  | 12.07s   | 0.922s   | **99.5%**   | 11,948        |
| timestamp_s | 80.0%  | 12.44s   | **1.230s** | **0.3%**  | 10,266        |
| truncate    | 75.0%  | 13.28s   | 0.936s   | 99.0%       | 13,036        |

---

## H1: Stable前缀 → 跨任务cache预热 ✅ 验证

**Turn-1 TTFT按任务顺序（stable策略）：**

```
Task 0:  1.534s  ← 冷启动，50个工具定义未被缓存
Task 1:  0.879s  ← 预热完成，下降43%
Task 2:  0.910s
Task 3:  0.889s
Task 4:  0.872s
...（此后所有任务稳定在 0.864-0.911s）
```

第一个任务完成后，~11K token的系统提示词+工具定义已被vLLM缓存。后续19个任务
的turn-1 TTFT全部受益，从1.534s降至~0.88s（降幅43%）。

**跨任务顺序执行**是这个效果成立的前提条件——这与Manus建议的"stable前缀"直接
对应：只要前缀不变，后续请求无论来自同一个任务还是不同任务，都能命中缓存。

---

## H2: 秒级时间戳 → 彻底摧毁Cache ✅ 强烈验证

**Turn-1 TTFT按任务顺序（timestamp_s策略）：**

```
Task 0:  1.156s
Task 1:  1.146s  ← 没有下降！
Task 2:  1.180s
Task 3:  1.243s
...（20个任务全部维持在 1.14-1.24s，无预热效果）
```

**稳定对比（stable vs timestamp_s）：**

```
Turn 1 - stable:      0.914s  ← 快
Turn 1 - timestamp_s: 1.183s  ← 慢 (+29%)

Turn 2 - stable:      0.897s
Turn 2 - timestamp_s: 1.281s  ← 慢 (+43%)

Cache hit rate:
  stable:      99.5%
  timestamp_s:  0.3%  ← 近乎完全失效
```

**机制**：`[Current time: 2026-03-22 18:23:50]` 这个字符串放在系统提示词开头，
仅6个token，但它导致其后全部 ~11K token 的缓存全部失效。vLLM无法复用任何
前缀块，每次请求都需要全量prefill 50个工具定义。

**实际代价**：TTFT从0.922s → 1.230s，增加0.308s（+33.4%）。对于高并发场景，
这意味着每个LLM call都多付出11K token的prefill计算代价，以及更高的token费用。

这直接验证了Manus文章的原话："单个token的差异就会导致其后所有内容的缓存失效"。

---

## H3: 截断历史 → Cache局部失效 + 准确率下降 ✅ 部分验证

### Cache影响（弱于预期）

```
截断策略 (keep_last_n=2):
  Turn 1: 0.901s (n=20)  ← 与stable相近 (0.914s)
  Turn 2: 0.903s (n=19)  ← 与stable相近 (0.897s)
  Turn 3: 1.054s (n=3)   ← 轻微上升 (stable: 1.027s)
```

Turn 3的TTFT轻微上升（1.054s vs 1.027s），但差异不显著（样本量仅n=3）。
原因：截断只影响对话部分（~500-1000 tokens），而缓存命中的主体是系统提示词+工具
定义（~11K tokens）。对大型prefill来说，截断引起的cache miss代价相对较小。

vLLM整体cache命中率：truncate=99.0% vs stable=99.5%，差异仅0.5%。

### 准确率影响（显著）

```
stable:   80% (16/20)
truncate: 75% (15/20)
```

差异来自 `chain4_temp_round_mod_abs` 这个任务：

- **stable**: 正确完成，步骤=4/4，耗时13.29s
- **truncate**: **失败**，步骤=10/4，耗时21.65s ← 陷入无限循环

truncate在这个任务上的TTFT轨迹：
```
[0.882, 1.026, 1.102, 1.092, 1.043, 1.094, 1.022, 1.037, 1.050, 1.046]
```
10轮！模型在turn 3开始截断上下文后，忘记了第一步的中间结果，不断重试，
直到达到max_turns=10上限。这是**截断历史导致的推理失败**的典型案例。

### 另一个延迟异常：chain5_temp_calc_mod_power_sumdig

- **stable**: 12.15s，ttft=[0.867, 1.033, 1.031, 1.028, 1.038, 0.877]
- **truncate**: 21.72s，ttft=[0.876, 1.031, 1.024, 1.018, 1.019, 0.874]

截断版本耗时近乎翻倍，但最终仍然得出正确答案——代价是更多的推理步骤。

## 关键发现

### 1. 跨任务的 "共享前缀" 效应存在且显著

vLLM的prefix cache不仅在同一任务的多轮之间生效，更重要的是**跨任务复用**。
50个工具定义（~11K tokens）一旦被第一个任务缓存，后续所有任务的turn-1都受益。
顺序执行（而非并行）是这个效应成立的关键。

### 2. 时间戳的代价被实测量化

| 代价 | 数值 |
|------|------|
| Cache命中率下降 | 99.5% → 0.3% |
| TTFT增加 | 0.922s → 1.230s (+33%) |
| 每LLM call额外prefill | ~11K tokens（冗余计算） |
| 对准确率影响 | 无（80% vs 80%） |

时间戳不影响模型的推理质量，但每次调用都需要重新prefill所有工具定义，
是纯粹的计算和成本浪费。

### 3. 截断历史的代价不在TTFT而在推理质量

与预期不同，截断上下文对TTFT的影响很小（系统提示词占主导）。
真正的代价是**推理链中断**：当模型需要引用更早的工具调用结果时，
截断导致它无法访问那些信息，可能陷入循环或做出错误判断。

### 4. 稳定前缀的"免费"收益

stable策略不需要任何特殊处理——仅仅通过保持系统提示词不变 + 顺序执行，
就获得了99.5%的cache命中率和跨任务TTFT从1.53s降至0.88s的效果。
这印证了Manus文章的核心观点：**稳定前缀本身就是最重要的优化**。

---

## 实验局限性

1. **样本量**: 每个策略仅1次run，无统计显著性检验
2. **turn 3+ 样本**: 仅有3-4个任务有3轮以上对话，截断的TTFT效应置信度低
3. **截断位置**: keep_last_n=2是合理默认值，但更激进的截断（n=1）可能显示更强信号
4. **截断发生时机**: 对话动态部分相对于11K token系统提示词很小，导致H3的cache效应较弱
5. **顺序vs并行**: 本实验专门设计为顺序执行以观察预热效果；生产场景的并行请求会有所不同

---

## 对 Manus 建议的量化验证

| Manus建议 | 实验验证 | 量化结果 |
|-----------|----------|----------|
| "避免在系统提示词中使用时间戳" | ✅ 强烈验证 | cache hit 99.5%→0.3%，TTFT +33% |
| "实现仅追加的上下文" | ✅ 部分验证 | 截断导致准确率-5%，异常任务耗时翻倍 |
| "稳定的共享前缀" | ✅ 验证 | 跨任务TTFT从1.53s→0.88s（task 0→1）|

在小规模（50工具，单节点vLLM）下，Manus的建议是正确的。在更大规模
（100+工具、多节点、高并发）场景下，这些代价会等比例放大。
