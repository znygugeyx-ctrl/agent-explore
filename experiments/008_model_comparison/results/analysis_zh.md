# 实验 008 分析：MiroThinker-1.7-mini vs Claude Haiku 4.5

## 总结

MiroThinker-1.7-mini（30B 推理模型，针对搜索任务微调）在 9 个 GAIA 任务上的准确率为 **16.7%**，
而 Claude Haiku 4.5 在 75 个 GAIA 任务上（007 基线）的准确率为 **67.3%**。
MiroThinker **参数量大 4 倍，但准确率差 4 倍**（在本实验条件下）。

## 失败模式分析

### 1. 上下文溢出（18 次中 9 次 = 50%）

模型的上下文窗口配置为 **32,768 token**（硬件限制：4× L40S 48GB，TP=4）。
MiroThinker 推荐配置为 **262,144 token**（256K）。这是主要的失败原因。

每次网页抓取（text_only 格式）会向上下文贡献约 5,000–15,000 token。
抓取 3–5 次后，32K 窗口即耗尽。上下文满后，vLLM 会拒绝新请求。

**证据**：所有高轮次/高搜索次数（20+ 轮、24+ 次搜索）的错误任务均为上下文溢出。
即便第一轮成功的任务，在第二轮也因更长的搜索路径或更长的思考链而失败。

### 2. 工具调用泄漏进最终答案（18 次中 4 次 = 22%）

模型使用了大量不在我的别名集中的工具名称：
- `search_webpage`、`search_query`（搜索变体）
- XML 解析器未能捕获这些变体 → 循环继续
- 最终达到 max_turns 上限，最后一条"无工具调用"的文本是不完整的 XML → 变成了答案

**观察到的模型工具名称多样性**：
```
google_search, search, search_web, web_search, search_webpage, search_query
fetch_page, fetch_webpage, scrape_webpage, browse
```

我的 `_SEARCH_TOOLS` 集合遗漏了：`search_webpage`、`search_query`。

### 3. 思考链消耗 Token 预算（18 次中 2 次）

由于该 vLLM 版本不支持 `--enable-reasoning`，思考 token 会占用与工具调用和最终答案相同的 `max_tokens=8192` 预算。对于土豆泥这类纯推理任务，模型生成了完整的 8192 token 思考链，却未能完成响应：
- 第一轮：不完整的思考内容泄漏为冗长答案
- 第二轮：`think_tokens=8320` → 思考被截断，答案为空

### 4. 格式指令不遵从

对于模型正确搜索并找到答案的任务，有时会输出冗长的推理段落，而非仅输出最终值。
"仅回答最终值"的指令被部分遵守，但并不稳定。

## 核心洞察

### MiroThinker 表现不佳的原因

**主要原因是基础设施不匹配，而非模型质量问题。**

1. **上下文窗口（32K vs 262K）**：MiroThinker 为长链多跳搜索工作流设计。
   模型卡明确建议使用 262K 上下文。受 GPU 内存限制，我们只用了 32K。
   在 32K 下，约半数尝试因上下文溢出而失败——这一瓶颈直接消除了模型的核心能力。

2. **思考 token 竞争**：未启用 `--enable-reasoning` 时，模型的 `<think>` 块
   与工具调用和最终答案共享同一 token 预算。在 32K 上下文、每轮 max_tokens=8192 的条件下，
   推理模型的工作记忆极为有限。

3. **工具名称多样性**：MiroThinker 的微调使用了多种工具名称。
   健壮的 MCP 集成需要映射所有变体（我们捕获了大部分，但并非全部）。

### 我们能得出的结论

- **当上下文充足时**，MiroThinker 表现良好：3 个成功任务（鱼袋容积、龙语文章单词、惠特尼博物馆书籍）均通过定向搜索正确作答。
- **延迟显著更高**（175s vs 76s），符合推理模型开销和更慢的生成速度（30B 模型约 20 tok/s vs Bedrock 的 Claude）。
- **模型 GAIA 基准声明的 82.7%** 是在完整 256K 上下文下测量的——与我们的 32K 设置不可比较。

### 更公平的对比需要

1. `max_model_len=131072` 或更高（需要 8× A100 80GB 或同等配置）
2. `--enable-reasoning` 支持，以将思考 token 与输出预算分离
3. 与 007 相同的任务集（本实验的 9 个任务与 007 的 75 个有部分重叠）

## 工程备注

- **请勿在本实验中将 MiroThinker 的数字与 Claude 进行比较**——上下文窗口约束使对比从根本上不公平。
- 本实验验证了：XML 工具调用解析可行、模型在上下文充足时能生成高质量推理、工具调用框架对非标准模型有良好扩展性。
- `restart_vllm.sh` 基础设施支持模型切换；使用更大上下文窗口重新运行（采用 8× GPU 实例，如 p4d.24xlarge 或 p3.16xlarge）是正确的下一步。
