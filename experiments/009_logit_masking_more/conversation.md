⏺ Logit Masking 背景调研

  一、技术定义

  LLM 每一步生成时，对词表中每个 token 输出一个 raw score（logit）。在 softmax 采样前可以干预：

  - Logit bias（软约束）：给特定 token 加偏移量 z_i' = z_i + bias，调节概率但不完全禁止
  - Logit masking（硬约束）：将 token logit 设为 -inf，概率归零，永远不会被采样
  - Guided decoding：用 FSM/CFG 追踪约束状态，每步动态计算合法 token 集合，mask 其余所有

  三、关键论文与观点

  1. Outlines (Willard & Louf, 2023)
  将文本生成建模为 FSM 状态转移。预计算词表索引（每个 FSM 状态 → 合法 token 集），per-step mask 开销极小。

  2. "Let Me Speak Freely?" (Tam et al., 2024)
  发现格式约束显著降低 LLM 推理能力。约束越紧，退化越严重。引发重大争议。

  3. dottxt 团队反驳
  指出 Tam 的实验 prompt 设计有缺陷（结构化 vs 非结构化用了不同 prompt）。用匹配 prompt 复现后，结构化生成在 GSM8K
  上反而提升 10%-70%。还发现约束降低了 prompt 格式敏感度（减少方差）。

  4. Coalescence 问题 (dottxt blog)
  不同 tokenization 路径产生相同字符串但不同概率分布。naive 的约束实现可能排除高概率序列。

  5. SGLang (Zheng et al., 2023)
  压缩 FSM + RadixAttention，6.4x 吞吐提升。"Jump-forward decoding" 跳过确定性 token 序列。

  ---
  四、Manus "Mask, Don't Remove"

  核心主张：不要在多轮对话中动态增减工具定义，而是保持 prompt 中工具列表固定，通过 logit mask 控制可选工具。

  理由：
  1. KV-cache 保护：工具定义在 prompt 前部，增删会使后续所有 token 的 cache 失效
  2. 上下文一致性：历史对话中有 tool_call: browser_click(...) 的记录，如果删掉 browser_click 的定义，模型会困惑
  3. 命名前缀技巧：所有浏览器工具以 browser_ 开头，命令行以 shell_ 开头，可以按前缀做 group-level mask，不需要
  stateful processor

  三种模式：auto（随意）、required（必须调工具）、specified（只能选子集）

  六、已知 tradeoff

  Constrained decoding 有帮助时：
  - 100% 结构正确性（免去 retry）
  - 部分场景提升推理（dottxt GSM8K 证据）
  - 降低 prompt 格式敏感度

  有害时：
  - Enum bias：我们实验 005/006 验证——guided decoding 导致 enum 字段准确率下降 25-75pp，因为 token
  概率分布在约束下被扭曲
  - 强制填充 hallucination：required 字段缺失信息时，模型被迫编造值（100% hallucination，所有 model size 一致）
  - Coalescence：不同 tokenization 路径的概率分布差异
  - 只适用于 self-hosted：OpenAI/Anthropic API 不暴露 logit，无法直接用
