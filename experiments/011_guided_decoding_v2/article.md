# 实验一：约束解码的得与失

约束解码（Constrained Decoding）是 logits masking 最常见的应用：给定一个 JSON Schema，在每一步解码时将不符合语法的 token 概率置零，强制输出合法 JSON。vLLM 的 xgrammar 和 OpenAI 的 Structured Output 都是这个思路。

问题很直接：在 prompt 里写清楚输出格式够不够用？还是必须上 logits masking 硬约束？这两种方式在格式保障和语义准确率上各有什么差异？

## 实验设计

**实验编号**：011（前置实验：005、006）
**模型系列**：Qwen3-0.6B / 1.7B / 4B / 8B / 14B / 32B
**总调用量**：7,200 次（6 模型 × 4 Track × 3 条件 × 100 任务）
**推理后端**：vLLM 0.18 + xgrammar，AWS g6e.12xlarge (4×L40S 48GB)

设计了 4 个结构化输出任务，schema 复杂度逐级递增：

| Track | 任务 | 数据集 | Schema | 任务数 |
|-------|------|--------|--------|-------|
| L1 | 数学推理 | GSM8K | `{"answer": int}` | 100 |
| L2 | 意图分类 | BANKING77 | `{"label": enum[77]}` | 100 |
| L3 | 实体识别 | Few-NERD | `{"entities": [{text, type: enum[8]}]}` | 100 |
| L4 | 食谱解析 | Recipe | 嵌套多字段（recipe_name, servings, ingredients[{food, qty, unit}], ...） | 100 |

从 L1 的单个整数字段，到 L4 的嵌套对象数组，覆盖了实际应用中常见的结构化输出需求。

模型使用 Qwen3 系列 6 个规模：0.6B、1.7B、4B、8B、14B、32B，推理后端 vLLM + xgrammar，温度 0.0。

两个核心条件：
- **prompt**：在 system prompt 中描述 schema，让模型自由输出 JSON
- **guided**：用 xgrammar 硬约束解码，保证输出严格符合 schema

另外加一组 prompt + thinking（允许 `<think>` 推理）作为能力上限参考。注意 guided + thinking 在技术上不可行——xgrammar 在第一个 token 就会阻断 `<think>` 标签。

为了同时评估格式和内容，每个输出按 2×2 矩阵分类：

|  | 内容正确 | 内容错误 |
|--|---------|---------|
| **格式有效** | Full Success | Semantic Failure |
| **格式无效** | Format Failure | Total Failure |

这样可以区分"答案对但格式错"和"格式对但答案错"，这是后面分析的基础。

总计 7,200 次调用。

## 格式保障的价值

先看 prompt 条件下（不使用约束解码），各模型能不能稳定输出合法的结构化数据。

**prompt 条件的结构有效率（%）：**

| 模型 | L1 GSM8K | L2 Banking77 | L3 Few-NERD | L4 Recipe |
|------|----------|-------------|-------------|-----------|
| 0.6B | 95 | 90 | 78 | **64** |
| 1.7B | **49** | 95 | 89 | **76** |
| 4B | 82 | 94 | 100 | 95 |
| 8B | 100 | 96 | 92 | 97 |
| 14B | 97 | 95 | 100 | 100 |
| 32B | 99 | 94 | 100 | 99 |

规律很明显：模型越小，schema 越复杂，格式失败率越高。0.6B 在 Recipe（L4）上只有 64% 的有效率——超过三分之一的输出无法解析。1.7B 在 GSM8K 上只有 49%，因为它倾向于输出自然语言的解题过程，而不是纯 JSON。

反过来，guided 条件下几乎所有组合的结构有效率都是 100%。唯一的例外是 1.7B 在 Banking77 上只有 69%，但这是 xgrammar 引擎处理 77 个 enum 值时超时导致的工程问题，不是语义问题。

可以为每个复杂度定义一个"格式稳定断点"——prompt 条件下结构有效率首次达到 95% 的最小模型规模：

| Schema 复杂度 | 格式稳定断点 |
|--------------|------------|
| L1 单字段 | 0.6B+ |
| L2 单 enum | 1.7B |
| L3 对象数组 | 4B |
| L4 嵌套多字段 | 4B ~ 8B |

schema 越复杂，需要的模型规模越大。之前实验（006）在简单 schema 上发现的 "0.6B→1.7B 格式断点"，在复杂 schema 上推移到了 4B 甚至 8B。

这就是约束解码的价值空间：在格式稳定断点以下的组合中，prompt 不够可靠，约束解码能把格式有效率直接拉到 100%。

## 语义准确率的代价

格式保障了，准确率会怎样？下面分任务对比 prompt 和 guided 的准确率。

**GSM8K — 数学推理（`{"answer": int}`）：**

| 模型 | prompt | guided | Δ |
|------|--------|--------|---|
| 0.6B | 3% | 1% | -2 |
| 1.7B | 63% | 9% | **-54** |
| 4B | 27% | 17% | -10 |
| 8B | 29% | 27% | -2 |
| 14B | 30% | 27% | -3 |
| 32B | 38% | 34% | -4 |

推理任务上，guided 全面低于 prompt。1.7B 最极端：prompt 条件下模型会在输出中夹带推理过程，格式虽然不规范（结构有效率只有 49%），但 63% 的任务答案是对的；guided 强制直接输出 `{"answer": N}`，砍掉了所有推理空间，准确率骤降到 9%。

**Banking77 — 77 类意图分类（`{"label": enum[77]}`）：**

| 模型 | prompt | guided | Δ |
|------|--------|--------|---|
| 0.6B | 33% | 41% | **+8** |
| 1.7B | 61% | 38%\* | -23 |
| 4B | 72% | 73% | +1 |
| 8B | 72% | 73% | +1 |
| 14B | 79% | 80% | +1 |
| 32B | 77% | 77% | 0 |

\*1.7B guided 有 31% 请求因 xgrammar 超时失败，记为错误。

分类任务上，4B 及以上模型 guided 和 prompt 几乎没有差异（±1pp）。0.6B 上 guided 反而高出 8pp——约束解码帮小模型把输出限制在合法的 77 个标签里，避免了输出无效标签。

**Few-NERD — 命名实体识别（`{"entities": [{text, type: enum[8]}]}`），Entity F1：**

| 模型 | prompt | guided | Δ |
|------|--------|--------|---|
| 0.6B | 0.064 | 0.112 | +0.048 |
| 1.7B | 0.428 | 0.444 | +0.016 |
| 4B | 0.456 | 0.532 | **+0.076** |
| 8B | 0.558 | 0.569 | +0.011 |
| 14B | 0.561 | 0.560 | -0.001 |
| 32B | 0.566 | 0.582 | +0.016 |

NER 任务上，guided 普遍略优。4B 的提升最明显（+0.076 F1），因为 xgrammar 约束实体类型必须在 8 个合法值中选择，减少了无效标签输出。

**Recipe — 食谱解析（嵌套多字段），准确率（相似度 ≥ 0.7）：**

| 模型 | prompt | guided | Δ |
|------|--------|--------|---|
| 0.6B | 10% | 11% | +1 |
| 1.7B | 13% | 22% | **+9** |
| 4B | 22% | 22% | 0 |
| 8B | 17% | 19% | +2 |
| 14B | 26% | 28% | +2 |
| 32B | 19% | 21% | +2 |

提取任务上，guided 一致地等于或略优于 prompt。1.7B 提升最大（+9pp），因为 prompt 条件下 1.7B 有 24% 的输出格式无效，guided 消除了这些损失。

**汇总来看**，约束解码的语义代价取决于任务类型：

- **推理任务**：一致有害。模型需要在输出过程中"想"，约束解码剥夺了这个空间。
- **分类任务**：大模型上中性（±1pp），小模型上有益（+8pp）。
- **提取任务（NER、Recipe）**：普遍中性或略正。约束解码帮模型更精确地落在合法输出空间里。

另外，之前小规模实验（005，30 个任务）发现 guided 在 enum 选择上存在系统性偏置（准确率从 100% 降到 60%）。但在这次 77 类 enum + 100 个任务的测试中，这个偏置没有复现——4B 以上模型 guided 和 prompt 准确率基本相同。005 的结论很可能是小样本 + 特定 enum 分布的 artifact。

## 何时值得用

上面分别看了格式保障（得）和语义代价（失）。把两者放在一起，用 Full Success 率（格式有效且内容正确）来衡量最终的可用输出：

**Full Success 率差值（guided − prompt，百分点）：**

| 模型 | L1 GSM8K | L2 Banking77 | L3 NER | L4 Recipe |
|------|----------|-------------|--------|-----------|
| 0.6B | -2 | **+9** | +2 | +1 |
| 1.7B | -15 | -23\* | 0 | **+9** |
| 4B | **+3** | +1 | **+8** | 0 |
| 8B | -2 | +1 | +3 | +2 |
| 14B | -1 | +1 | +1 | +2 |
| 32B | -4 | +2 | 0 | +2 |

\*1.7B Banking77 受 xgrammar 超时影响。

GSM8K（推理任务）列几乎全是负数——约束解码在这类任务上始终得不偿失。但在分类和提取任务上，情况不同：

- 0.6B + Banking77：guided Full Success 41% vs prompt 32%，差 9 个百分点
- 1.7B + Recipe：guided 22% vs prompt 13%，差 9 个百分点
- 4B + NER：guided 31% vs prompt 23%，差 8 个百分点

一个有意思的案例是 4B + GSM8K。前面提到 guided 的准确率比 prompt 低 10pp（17% vs 27%），看起来是明显的语义退化。但从 Full Success 看，guided 反而高出 3pp（17% vs 14%）。原因是 prompt 条件下 4B 在 GSM8K 上有 18% 的结构无效输出——其中 13% 的任务答案本身是对的，但因为格式问题无法使用。guided 消除了这些格式损失，虽然准确率下降了，但可用输出反而更多。

之前实验 005 得出的结论是"约束解码只是可靠性增强器，不是能力增强器"——它保障格式，但不会提升准确率。这个结论在大模型 + 简单 schema 上仍然成立。但在小模型 + 复杂 schema 上被推翻了：当模型的格式能力不足时，约束解码不仅保障了格式，还实质性地提升了最终可用输出。原因很直接: 模型"知道"答案但表达不出来，约束解码帮它跨过了格式门槛。

## 总结

约束解码通过 logits masking 强制结构化输出，它的适用范围可以用一个矩阵概括：

|  | 推理任务 | 分类 / 提取任务 |
|--|---------|---------------|
| **小模型（≤ 4B）** | 有害 | 推荐 |
| **大模型（8B+）** | 有害 | 中性 |

推理任务需要模型在输出过程中思考，约束解码剥夺了这个空间，在所有规模上都是负面的。分类和提取任务的答案空间本身有限，约束解码在小模型上是净正面选择（格式保障 + 准确率提升），在大模型上接近零收益（prompt 已经够可靠）。
